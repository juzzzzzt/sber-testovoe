# Task 3: Система валидации агента генерации презентаций

## Цель

Спроектировать систему оценки качества агента из Task 2, которая позволяет:
1. Измерять качество каждого компонента пайплайна (retrieval, generation, pptx).
2. Сравнивать разные версии агента (промпт, retrieval-логика, параметры поиска, чанкинг).
3. Отвечать на вопрос: **стало ли лучше или хуже** после изменения.

---

                           │

## Тестовый набор (Test Set)

### Формирование
- **8 клиентов** из `clients.json` (РЖД, Аэрофлот, Норникель, Магнит, Лукойл, МТС, ФосАгро, РусГидро).
- Для каждого клиента вручную размечается **ground truth** по трём осям:

| Ось | Что размечается | Источник |
|---|---|---|
| **Продукты** | Список релевантных продуктов (2–4 шт.) + список нерелевантных | `internal_notes.json` |
| **Новости** | 3–5 релевантных новостей из `news.csv` | ручная разметка по `client_name` |
| **Профиль** | 3 ключевых риска/темы из профиля | выжимка из `clients.json` |

### Пример записи в тестовом наборе

```json
{
  "client": "РЖД",
  "ground_truth": {
    "relevant_products": ["IRS", "Repo", "Deposit"],
    "irrelevant_products": ["Commodity Hedge", "Structured Note"],
    "relevant_news_ids": [12, 45, 78],
    "key_risks": ["рост процентных ставок", "валютные платежи по оборудованию", "кассовые разрывы"]
  }
}
```

### Размер
- 8 клиентов × 1 запрос = **8 тестовых кейсов** (для прототипа достаточно).
- Для статистической значимости можно расширить до 50+ через аугментацию запросов («подготовь презентацию», «расскажи про риски», «какие продукты предложить»).

## Метрики качества

Метрики разбиты по компонентам пайплайна — это важно, потому что улучшение retrieval может ухудшить generation, и нужно видеть, где именно произошёл сдвиг.

### 1. Retrieval-метрики (качество поиска)

| Метрика | Формула / смысл | Что измеряет |
|---|---|---|
| **Precision@k** | `|relevant ∩ retrieved_k| / k` | Сколько из топ-k найденных документов релевантны |
| **Recall@k** | `|relevant ∩ retrieved_k| / |relevant|` | Какая доля релевантных документов найдена |
| **MRR** | `1 / rank_first_relevant` | Насколько высоко первый релевантный документ |
| **NDCG@k** | DCG / IDCG | Учитывает порядок релевантных документов |

**Применение:** оцениваем поиск продуктов, заметок и новостей отдельно.

### 2. Content-метрики (качество генерации текста)

| Метрика | Как считается | Что измеряет |
|---|---|---|
| **Relevance score** | LLM-as-judge: «Насколько ответ релевантен запросу?» (1–5) | Смысловое соответствие |
| **Factual consistency** | NLI-модель: entails/contradicts между источником и текстом | Галлюцинации |
| **ROUGE-L** | Пересечение longest common subsequence с ground truth | Покрытие ключевых фраз |
| **BERTScore** | Косинусная близость эмбеддингов | Семантическая близость |

### 3. Business-метрики (специфично для нашего агента)

| Метрика | Формула | Что измеряет |
|---|---|---|
| **Product match accuracy** | `|recommended ∩ relevant| / |recommended|` | Доля правильно подобранных продуктов |
| **Note adherence score** | LLM-as-judge: «Следует ли ответ внутренней заметке?» (1–5) | Соблюдение бизнес-логики |
| **False positive rate** | `|recommended ∩ irrelevant| / |recommended|` | Доля нерелевантных продуктов в выдаче |

**Это ключевые метрики для Task 2** — они показывают, правильно ли агент интерпретирует внутренние заметки.

### 4. PPTX-метрики (качество файла)

| Метрика | Как считается |
|---|---|
| **Structural validity** | Все ли слайды на месте, нет ли пустых полей |
| **Text overflow** | Превышает ли текст границы слайда (через `python-pptx` inspection) |
| **Readability score** | Flesch-Kincaid по текстам слайдов |

### 5. Итоговая оценка

**Composite score** = взвешенная сумма:
```
Score = 0.3 × retrieval + 0.3 × business + 0.25 × content + 0.15 × pptx
```
Веса настраиваются под бизнес-приоритеты.

## Способ сравнения версий

### 1. Версионирование
Каждая версия агента описывается конфигом:
```yaml
version: "v1.2"
changes: "switched retrieval from BM25 to hybrid k-NN + BM25"
params:
  retrieval:
    strategy: hybrid
    top_k: 5
    knn_weight: 0.7
  generation:
    temperature: 0.3
    max_tokens: 2000
```

### 2. Запуск
Для каждой версии прогоняем весь тестовый набор → получаем таблицу метрик.

### 3. Отчёт сравнения

| Метрика | v1.0 (baseline) | v1.1 (hybrid search) | v1.2 (new prompt) | Δ v1.1→v1.2 |
|---|---|---|---|---|
| Product match accuracy | 0.62 | 0.75 | 0.81 | **+0.06** ✅ |
| Note adherence | 3.2 | 4.1 | 4.5 | **+0.4** ✅ |
| Retrieval Recall@3 | 0.58 | 0.71 | 0.69 | -0.02 ⚠️ |
| Factual consistency | 0.88 | 0.85 | 0.91 | **+0.06** ✅ |
| Text overflow (pptx) | 0.15 | 0.12 | 0.08 | **-0.04** ✅ |
| **Composite** | **0.68** | **0.76** | **0.82** | **+0.06** ✅ |

### 4. Статистическая значимость
- Для числовых метрик — **paired t-test** или **bootstrap** по 8 клиентам.
- Порог: p < 0.05 считаем улучшение значимым.
- Для LLM-as-judge (ordinal 1–5) — **Wilcoxon signed-rank test**.

### 5. Автоматизация
Пайплайн валидации запускается одной командой:
```bash
python validate.py --config v1.2.yaml --test-set test_set.json --output report_v1.2.md
```
На выходе — markdown-отчёт с таблицей и дельтами относительно baseline.

## Что даёт такая система

1. **Локализация проблем**: если упал composite score, видно, какой компонент виноват (retrieval / generation / pptx).
2. **Защита от регрессий**: перед мержем новой версии прогоняем тесты — если значимое ухудшение по ключевым метрикам, мерж блокируется.
3. **Data-driven решения**: вместо «мне кажется, стало лучше» — конкретные цифры и статистика.
4. **Итеративное улучшение**: можно тестировать гипотезы (другая модель эмбеддингов, другой промпт, другой top-k) и выбирать лучшую по composite score.

## Стек реализации (если потребуется)

- **Test set**: JSON с ground truth, версия-controlled.
- **Evaluators**:
  - Retrieval: `ranx` или `ir-measures`.
  - Content: `rouge-score`, `bert-score`, NLI-модель (например, `cross-encoder/nli-deberta-v3-base`).
  - LLM-as-judge: GigaChat сFew-shot промптом и структурированным выводом.
  - PPTX: кастомные проверки через `python-pptx`.
- **Отчёты**: `pandas` + markdown-шаблоны или `streamlit`-дашборд.
- **CI**: GitHub Actions запускает валидацию на каждый PR.

## Вывод

Система валидации превращает разработку агента из «творчества» в **инженерный процесс**: каждое изменение промпта, retrieval-стратегии или параметров генерации оценивается количественно, сравнивается с baseline и проверяется на статистическую значимость. Это позволяет итеративно улучшать агента и уверенно говорить «стало лучше», подкрепляя утверждение цифрами.
